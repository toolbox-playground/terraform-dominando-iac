# 03_05 - Init

## O que faz

`terraform init` **prepara o diretório** para que outros comandos (`plan`, `apply`, etc.) possam rodar. É o primeiro comando que você roda em um diretório novo.

```bash
terraform init
```

Saída típica (primeira vez):

```text
Initializing the backend...
Initializing provider plugins...
- Finding hashicorp/aws versions matching "~> 5.0"...
- Installing hashicorp/aws v5.40.0...

Terraform has been successfully initialized!
```

## O que ele faz exatamente

### 1. Configura o backend

Lê o bloco `terraform { backend "xxx" { ... } }` e **inicializa** o backend:

- Se é `local`: cria o arquivo `terraform.tfstate` no momento do primeiro apply.
- Se é remoto (S3, GCS, Azure, HCP): valida conectividade, verifica state existente, oferece migração se houver state local.

### 2. Baixa providers

Lê `required_providers` e baixa os binários dos providers para `.terraform/providers/`:

```text
.terraform/
└── providers/
    └── registry.terraform.io/
        └── hashicorp/
            └── aws/
                └── 5.40.0/
                    └── darwin_arm64/
                        └── terraform-provider-aws_v5.40.0_x5
```

### 3. Gera/atualiza o lock file

Cria/atualiza `.terraform.lock.hcl` com **hashes exatos** dos providers baixados. Este arquivo deve ser **commitado** para garantir reprodutibilidade entre máquinas e entre CI.

Exemplo de lock:

```hcl
provider "registry.terraform.io/hashicorp/aws" {
  version     = "5.40.0"
  constraints = "~> 5.0"
  hashes = [
    "h1:abc123...",
    "zh:def456...",
  ]
}
```

### 4. Baixa módulos externos

Se você usa `module "x" { source = "..." }` apontando para git, HTTP ou Registry, o `init` baixa/clona esses módulos para `.terraform/modules/`.

### 5. Configura workspaces

Se o backend suporta workspaces, cria estrutura inicial (workspace `default`).

## Quando (re)rodar

- **Primeira vez** no diretório.
- Após **adicionar/remover providers** em `required_providers`.
- Após **mudar versão** de provider.
- Após **adicionar/modificar módulos** externos.
- Após **mudar backend** (ex.: de local para S3).
- Ao clonar o repo em uma máquina nova.

Se você só mexe em recursos sem mudar providers, **não precisa** re-rodar `init`. Mas também não faz mal.

## Flags úteis

| Flag | Uso |
|------|-----|
| `-upgrade` | Atualiza providers para a versão mais nova permitida pelas constraints. Sem isso, o init fica preso na versão do lock. |
| `-reconfigure` | Reconfigura o backend ignorando config anterior. Usa quando muda backend. |
| `-migrate-state` | Migra state entre backends (de local para S3, por exemplo). |
| `-backend-config=FILE` | Carrega config do backend de um arquivo (bom para separar segredos). |
| `-backend-config="k=v"` | Config do backend via linha de comando. |
| `-backend=false` | Não configura backend (útil em CI para validar sem credenciais). |
| `-input=false` | Não prompta; CI. |
| `-no-color` | Sem cores; CI. |
| `-lockfile=readonly` | Não atualiza o lock. Em CI para garantir reprodutibilidade. |
| `-plugin-dir=DIR` | Usa providers de um diretório local em vez de baixar (air-gap). |

## Exemplos

### Init simples

```bash
terraform init
```

### Upgrade providers

```bash
terraform init -upgrade
```

### Backend com config via arquivo

```hcl
# backend.hcl
bucket = "meu-tf-state"
key    = "prod/terraform.tfstate"
region = "us-east-1"
```

```bash
terraform init -backend-config=backend.hcl
```

Útil quando o mesmo código Terraform é usado em múltiplos ambientes (dev, staging, prod), cada um com seu backend.

### Init em CI sem credenciais da nuvem

```bash
terraform init -backend=false
terraform validate  # rodando sem backend
```

Valida código em um job sem precisar de credenciais.

### Migrando de state local para S3

1. Começa com `backend "local"` e recursos já aplicados.
2. Muda para `backend "s3"`.
3. Roda:

```bash
terraform init -migrate-state
```

Terraform pergunta se quer copiar o state local para o S3. Responde `yes`. Done.

## Estrutura gerada

Depois do `init`:

```text
meu-projeto/
├── main.tf
├── versions.tf
├── .terraform/              # NÃO commitar
│   ├── providers/
│   └── modules/
└── .terraform.lock.hcl      # COMMITAR
```

## Cuidados

- **Não commite `.terraform/`** (tamanho grande; depende de sistema).
- **Commite `.terraform.lock.hcl`** (pequeno; garante reprodutibilidade).
- **Trate `init` como idempotente** — rodá-lo duas vezes não causa problemas.
- **Em CI, sempre use `-lockfile=readonly`** para impedir que o CI atualize o lock silenciosamente.

## Troubleshooting

### "Error: Failed to install provider"

- Sem internet? Provider privado em registry?
- Verifique `TF_PLUGIN_CACHE_DIR` se usa cache.
- Tente `terraform init -upgrade`.

### "Error: Backend configuration changed"

- Você mudou o bloco `backend`. Use `-reconfigure` (se quer resetar) ou `-migrate-state` (se quer mover state).

### "Error: Inconsistent dependency lock file"

- Lock file tem hash que não bate com o provider baixado. Rode `terraform init -upgrade` ou apague `.terraform/` e reinit.

### Init super lento

- `.terraform/providers/` já é grande? Configure **plugin cache**:

  ```bash
  export TF_PLUGIN_CACHE_DIR="$HOME/.terraform.d/plugin-cache"
  mkdir -p "$TF_PLUGIN_CACHE_DIR"
  ```

  Providers compartilhados entre projetos.
