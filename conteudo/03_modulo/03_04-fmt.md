# 03_04 - FMT

## O que faz

`terraform fmt` **reescreve** seus arquivos `.tf` aplicando o estilo oficial da HashiCorp:

- 2 espaços de indentação.
- Alinhamento de `=` em blocos de atributos.
- Quotes padronizadas.
- Espaçamento em volta de operadores.
- Remoção de interpolações redundantes (`"${var.x}"` → `var.x`).

```bash
terraform fmt
```

Saída: nome dos arquivos alterados (ou vazio se nada mudou).

## Por que importa

- **Code review sem ruído**: diffs mostram só mudanças de lógica, não de whitespace.
- **Padrão único de time**: ninguém discute "espaço antes ou depois do `=`".
- **Leitura mais rápida**: cérebro pega padrões quando tudo é consistente.
- **Ferramentas (tflint, tfsec) funcionam melhor** sobre código formatado.

## Flags úteis

| Flag | O que faz |
|------|-----------|
| (sem flag) | Formata arquivos no diretório atual e imprime quais foram alterados. |
| `-recursive` | Desce em subdiretórios (essencial em repos grandes). |
| `-check` | Não altera arquivos; só retorna exit code 0 se tudo já está formatado, 3 se algo precisa ser ajustado. |
| `-diff` | Mostra o diff do que seria alterado. |
| `-write=false` | Imprime o conteúdo formatado em stdout em vez de escrever. |
| `-list=false` | Não imprime nomes de arquivos alterados. |
| `-no-color` | Remove cores. |

## Exemplos

### Formatar tudo, recursivamente

```bash
terraform fmt -recursive
```

### Ver o que seria alterado sem mexer

```bash
terraform fmt -recursive -check -diff
```

Em CI, esse é o uso canônico para **falhar o build** se alguém esqueceu de rodar `fmt`:

```bash
terraform fmt -check -recursive -diff
# exit code 3 se precisa formatar
```

### Antes/depois

**Antes:**

```hcl
resource "aws_instance" "web"{
ami = "ami-0123"
   instance_type="t3.micro"
    tags={
Name="web"
}
}
```

**Depois de `terraform fmt`:**

```hcl
resource "aws_instance" "web" {
  ami           = "ami-0123"
  instance_type = "t3.micro"
  tags = {
    Name = "web"
  }
}
```

## Integração com editor

No **VS Code** com a extensão **HashiCorp Terraform**:

```json
{
  "editor.formatOnSave": true,
  "[terraform]": {
    "editor.defaultFormatter": "hashicorp.terraform"
  },
  "[terraform-vars]": {
    "editor.defaultFormatter": "hashicorp.terraform"
  }
}
```

Com isso, cada `Ctrl+S` / `Cmd+S` formata automaticamente. Você nunca mais precisa rodar `fmt` manualmente localmente.

## Pre-commit hook

```yaml
repos:
  - repo: https://github.com/antonbabenko/pre-commit-terraform
    rev: v1.86.0
    hooks:
      - id: terraform_fmt
```

Com `pre-commit install`, qualquer commit passa por `fmt` automaticamente — e se modificar algum arquivo, o commit falha e pede para adicionar as alterações.

## Pipeline CI

```yaml
name: Terraform CI
on: pull_request

jobs:
  terraform-fmt:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: hashicorp/setup-terraform@v3
      - name: Check terraform fmt
        run: terraform fmt -check -recursive -diff
```

Esse job **falha** se alguém esqueceu de formatar e **mostra o diff** do que deveria ter sido.

## Relação com `validate`

- `fmt` altera **formatação**, nunca semântica.
- `validate` não reformata, só reporta erros.
- Combinar os dois: `fmt -check && validate`.

## O que `fmt` **não** faz

- Não ordena atributos por nome (você decide a ordem).
- Não reordena blocos (`resource`, `data`, etc.).
- Não adiciona `description` em variáveis faltantes.
- Não corrige erros de sintaxe (só funciona em HCL válido).
- Não aplica estilos além do oficial (não dá para configurar).

Se você quiser regras customizadas, combine com **tflint**.
