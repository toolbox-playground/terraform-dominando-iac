# 01_05 - Toil

## Definição

O termo **toil** foi formalizado pela engenharia de SRE do Google no livro *Site Reliability Engineering* (O'Reilly, 2016). Traduzindo de forma livre, é o **trabalho operacional chato, repetitivo e sem valor duradouro** que um time de engenharia é obrigado a fazer para manter os sistemas rodando.

Não é todo trabalho manual que é toil. Engenharia de verdade (projetar arquitetura, escrever código novo, corrigir bug) pode ser manual e não ser toil. O que define toil são as características abaixo.

## Características do toil (segundo o Google SRE)

Um trabalho é toil quando combina **várias** dessas propriedades:

1. **Manual** — Exige humano para executar passo a passo.
2. **Repetitivo** — É feito de novo e de novo, do mesmo jeito.
3. **Automatizável** — Uma máquina poderia fazer (o problema não é conceitual, é só "ainda não automatizamos").
4. **Reativo em vez de estratégico** — Atende demanda imediata, não constrói valor de longo prazo.
5. **Sem valor duradouro** — Ao terminar, o sistema voltou ao estado anterior. Não evoluiu.
6. **Escala linear** (ou pior) com o serviço — Mais tráfego/mais clientes = mais toil.

Se um trabalho marca 4+ dos critérios acima, é toil.

## Exemplos de toil no dia a dia

- Criar manualmente bucket S3 para cada novo projeto via console.
- Dar permissões IAM clicando na AWS Console toda vez que alguém entra no time.
- Rodar o mesmo script de "setup de ambiente" para cada engenheiro novo.
- Escalar manualmente uma frota de EC2 quando o tráfego aumenta.
- Rotacionar certificados TLS manualmente a cada 90 dias.
- Aplicar patches de segurança máquina por máquina via SSH.
- Criar e devolver acessos a banco de dados em cada projeto novo.
- Responder ao alerta "disco cheio" esvaziando `/tmp` semana sim, semana não.

## Por que toil é ruim

- **Consome tempo que poderia ser engenharia**. Google SRE estabelece **limite máximo de 50% do tempo em toil**; o resto deve ser projeto.
- **Causa burnout**. Repetição sem valor desmotiva e afasta bons engenheiros.
- **Introduz erros**. Humanos cansados esquecem passos, invertem parâmetros, copiam valores errados.
- **Não escala**. Se cada novo cliente/serviço gera 1h de toil, com 100 clientes você tem um time inteiro só respondendo a toil.
- **Esconde o custo**. Toil raramente aparece em relatórios; aparece em deadlines perdidos.

## Como IaC e Terraform reduzem toil

A regra geral: **se dói repetir, automatize**.

**Antes (toil):**
> Toda vez que um novo time entra, alguém da plataforma abre 15 serviços na AWS para criar VPC, subnets, security groups, roles, buckets, DNS — leva meio dia e sempre tem algum passo esquecido.

**Depois (Terraform):**
```hcl
module "ambiente_novo_time" {
  source = "git::git@github.com:minha-org/tf-modules.git//ambiente-time?ref=v2.1.0"

  nome_time       = "checkout"
  ambiente        = "producao"
  permissoes_role = ["s3:ReadWrite", "sqs:*"]
}
```

Cinco linhas, `terraform apply`, 3 minutos. E o ambiente sai idêntico ao de todos os outros times, sem erro humano, com auditoria no Git.

## Métricas de toil

Times maduros medem toil para justificar investimento em automação:

- **% do tempo semanal gasto em toil** (auto-report dos engenheiros).
- **Toil por serviço** — serviço que gera mais toil é candidato a refatoração.
- **Alertas recorrentes** — mesmo alerta disparando toda semana é toil disfarçado de incidente.
- **Runbooks executados manualmente** — se um runbook é rodado todo mês, transforme em Terraform/pipeline.

O livro *The Site Reliability Workbook* (2018) detalha técnicas para medição.

## Diferença entre toil e "trabalho operacional legítimo"

Nem todo trabalho operacional é toil:

- **Toil**: criar manualmente a VPC para o time X (já criei 20 iguais antes).
- **Não-toil**: investigar por que a latência do serviço Y subiu de 50ms para 500ms (problema novo, raciocínio, valor em entender).

**Toil**: responder ao alerta de disco cheio pela milésima vez com `rm /tmp/*`.
**Não-toil**: escrever o playbook que roda essa limpeza automaticamente e remove o alerta para sempre.

O trabalho que **elimina** toil é engenharia de verdade. O trabalho que só **mitiga temporariamente** é toil.

## Relação com os outros conceitos do módulo

- **IaC** é o principal antídoto ao toil de provisionamento.
- **Infra imutável** elimina toil de patching manual e correção de drift.
- **Idempotência** permite que a automação seja segura — sem ela, o medo de rerrodar mantém o toil vivo.

IaC + imutabilidade + idempotência é a combinação que transforma "1h de toil por cliente" em "1 linha de código, zero toil por cliente".
